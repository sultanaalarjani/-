# لوحة "إدارة عمليات الأداء" — مستند تقني للنشر (لتقنية المعلومات)

## ١) ما هو التطبيق؟
تطبيق ويب لقياس أداء القطاعات عبر مؤشرات استراتيجية (المستهدف مقابل المحقق لكل ربع)،
بواجهة رسومية (عدّادات وبطاقات)، ونظام صلاحيات، وتسجيل دخول آمن برقم الجوال عبر رمز (OTP).

- **مدير الإدارة (Admin):** يرى كل القطاعات، يدير الهيكل والمستخدمين والعتبات.
- **مدير القطاع (Manager):** يدخل بيانات قطاعاته المسندة إليه فقط.

## ٢) البنية التقنية (كيف هو مبني)
| العنصر | التقنية |
| --- | --- |
| الإطار | **Next.js 15 (App Router)** — يجمع الواجهة والـ API في تطبيق واحد |
| اللغة | **TypeScript / React 19** |
| الرسوم/العدّادات | **SVG** مخصّص (بدون خدمات خارجية) |
| التخزين | **PostgreSQL** (يُفضّل) عبر `DATABASE_URL`، أو **ملف JSON محلي** كاحتياط |
| تسجيل الدخول | **OTP برقم الجوال**، وجلسات بكوكي `httpOnly` |
| إرسال الرمز | بوابة SMS قابلة للتهيئة (مزوّد عام أو Twilio)، مع وضع تجربة |
| التشغيل | **Docker** (موصى به) أو **Node.js 20+** مباشرة |

> بنية أحادية (Monolith) بسيطة: لا يوجد Backend منفصل ولا خدمات معقّدة. الـ API مسارات
> داخل Next.js تحت `/api/*`.

## ٣) نموذج البيانات
- **Sectors (قطاعات):** حتى 7.
- **Indicators (مؤشرات):** 9 افتراضية، قابلة للإضافة/التعديل، لكل مؤشر وحدة (نسبة/عدد).
- **Periods (فترات):** أرباع السنة.
- **Measurements (قياسات):** لكل (قطاع × مؤشر × ربع) قيمتان: `target` و`actual`.
- **Users:** `phone`, `role` (admin/manager), `sectorIds`, `active`.
- **Settings:** عتبات الحالة (goodThreshold / excellentThreshold).

في Postgres يُخزَّن كامل النموذج كوثيقة **JSONB** في جدول `app_state` يُنشأ تلقائيًا — لا حاجة
لتهيئة سكيمة يدويًا.

## ٤) المتطلبات
- **Docker + Docker Compose** (الأسهل)، أو **Node.js 20+**.
- **PostgreSQL** (للحفظ الدائم) — يمكن استخدام سيرفر Postgres داخلي، أو Postgres سحابي.
- منفذ TCP **3000** (قابل للتغيير عبر `PORT`).

## ٥) متغيّرات البيئة
| المتغير | الوصف |
| --- | --- |
| `ADMIN_PHONE` | جوال مدير الإدارة الأول (يُنشأ تلقائيًا عند أول تشغيل) |
| `ADMIN_NAME` | اسمه (اختياري) |
| `DATABASE_URL` | رابط اتصال Postgres للحفظ الدائم. إن تُرك فارغًا يُستخدم ملف محلي |
| `COOKIE_SECURE` | `true` خلف HTTPS، `false` للتشغيل الداخلي عبر HTTP |
| `SEED_DEMO` | `false` للاستخدام الفعلي (بدون بيانات تجريبية) |
| `SMS_PROVIDER` | `generic` أو `twilio` (فارغ = وضع تجربة يعرض الرمز في السجل) |
| `SMS_API_URL`, `SMS_API_KEY`, `SMS_SENDER` | لمزوّد SMS عام (POST JSON) |
| `TWILIO_SID`, `TWILIO_TOKEN`, `TWILIO_FROM` | لمزوّد Twilio |

## ٦) النشر — Docker (موصى به)
```bash
cp .env.example .env      # ثم عبّئ المتغيّرات أعلاه
docker compose up -d --build
```
- يعمل تلقائيًا ويعيد التشغيل بعد إعادة الإقلاع (`restart: unless-stopped`).
- إذا لم تُستخدم قاعدة Postgres، تُحفظ البيانات في فوليوم `dashboard_data` (قرص دائم).
- الأفضل: ضبط `DATABASE_URL` لقاعدة Postgres داخلية لضمان الحفظ والنسخ الاحتياطي المؤسسي.

## ٧) النشر — بدون Docker (Node)
```bash
npm install
npm run build
npm start                 # يستمع على http://0.0.0.0:3000
```
للإبقاء عليه دائمًا: `pm2 start "npm start" --name perf-dashboard && pm2 save`.

## ٨) بوابة الرسائل (SMS)
- **مزوّد عام:** `SMS_PROVIDER=generic` + `SMS_API_URL` + `SMS_API_KEY` + `SMS_SENDER`.
  (الطلب POST JSON بالحقول: `to`, `sender`, `message`.)
- **Twilio:** `SMS_PROVIDER=twilio` + `TWILIO_SID/TOKEN/FROM`.
- بدون تهيئة: وضع تجربة يطبع الرمز في السجل بدل إرساله.
- **إن وُجدت بوابة رسائل حكومية معتمدة لديكم، يمكن ربطها عبر المزوّد العام مباشرة.**

## ٩) الأمان والنسخ الاحتياطي
- الدخول مقصور على الأرقام التي يضيفها مدير الإدارة (OTP).
- كوكي جلسة `httpOnly`؛ فعّلوا `COOKIE_SECURE=true` خلف HTTPS.
- ضعوه خلف **Nginx/Caddy** مع شهادة HTTPS للنشر الرسمي.
- **النسخ الاحتياطي:** نسخ احتياطي دوري لقاعدة Postgres (أو لفوليوم `dashboard_data` عند استخدام الملف).
